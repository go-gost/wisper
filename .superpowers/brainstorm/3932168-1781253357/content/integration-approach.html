Inspector Backend Integration — How?

How should wisper's Go backend provide inspector data to the frontend?

A

Import inspector/pkg into wisper

Add github.com/go-gost/inspector to wisper's go.mod. Reuse inspector's pkg/model (MongoDB queries, Redis hub) and pkg/api (handlers) directly. Wire into wisper's existing HTTP mux.

Pros

  • Zero logic duplication
  • Bug fixes in inspector propagate automatically
  • Same data structures as the React SPA

Cons

  • Adds MongoDB + Redis deps to wisper
  • Inspector uses Gin, wisper uses net/http — needs adapter
  • Inspector code is singleton-heavy (global config, global DB clients)
B

Reverse proxy to inspector backend

Run inspector as a sidecar or separate service. Wisper acts as a reverse proxy: /api/tunnels/{id}/records → inspector:8000/api/records/query?client_id={id}. Wisper stays thin, inspector runs independently.

Pros

  • Wisper stays dependency-free
  • Inspector runs independently, can be shared across wisper nodes
  • Simplest implementation — a few lines of proxy code

Cons

  • Requires deploying two services
  • Extra network hop on every query
  • Authentication/authorization across services
C

Extract & embed — copy model logic into wisper

Copy the essential inspector model types + query logic (~3 files) into a new wisper/inspector/ Go package. Strip the Gin dependency, adapt to wisper's net/http patterns. Keep MongoDB + Redis connections scoped to wisper's config.

Pros

  • Clean integration — matches wisper's existing patterns
  • No Gin dependency, no singleton globals
  • Full control over API surface and auth

Cons

  • Forks inspector logic (drift over time)
  • Still adds MongoDB + Redis deps
  • Duplicates ~500 lines of model code
